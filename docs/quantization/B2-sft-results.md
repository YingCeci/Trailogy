# SFT'd Gemma 4 E2B — MLX (iOS-deployable) quantization baselines

## TLDR

Per-variant MLX results on the data-aug-enwiki SFT — the actual iOS deploy candidates. bf16 ceiling (M0) is 88.3 % PlantNet at n=300. Flat affine (M1/M2/M3 at g128/g64/g32) lands 83-84 % at 3.2-3.6 GB. EoRA post-quant adapter on top of M2 recovers most of the drop: M8b r=64 at 3.6 GB / 88.0 % is within statistical noise of bf16 and is the recommended ship. MLX hybrid GPTQ/AWQ/DWQ cores are blocked or worse on Gemma 4.

Per-variant detail for every MLX-side quantization we've run on our
**SFT'd** Gemma 4 E2B. These are the candidates that load directly via
`mlx_vlm.load` and `mlx-swift-lm` on iOS. HF/CUDA variants live in
`B1-sft-results.md`. Public 4-bit MLX references on
the un-SFT'd base live in `01-baselines.md`.

## Pass criteria

For any MLX variant against its own bf16 reference on the same split:

- **PlantNet drop ≤ 5 pts vs bf16 reference (M0)** → ✅
- **PlantNet drop 5-10 pts** → 🟡 (investigate per-method)
- **PlantNet drop > 10 pts** → ⚠ trip — likely scope error (vision
  tower silently quantized, or projector regression). Run
  `inspect_vision_dtype` tripwire (`src/quantization/scripts/inspect/vision_dtype.py`).

## Per-variant detail

### M0 — `bf16_r0_sft_aug_enwiki` ✅

- Loader: `mlx_vlm.load` directly on the merged bf16 dir. Required a
  one-shot sidecar (`prep_inject_kv_shared.py`) injecting zero-init
  K/V/k_norm tensors for layers 15-34 — the KV-shared region. HF
  transformers correctly omits these (new format); mlx_vlm's
  `Attention.__init__` allocates them unconditionally and `load_weights`
  errors with "Missing 60 parameters" otherwise. The zero tensors are
  inert at runtime (KV-shared layers fetch K/V from cache state per
  `mlx_vlm/models/gemma4/language.py:204-220`).
- species_match = **88.3 %** (265/300); ROUGE-L mean = 0.856
- Eval wall: 290 s on M5 Pro 32 GB.
- This is the ceiling for any MLX variant on this exact (model,
  sorted test/ split, n=300 seed=0) combination.
- **Cross-validation (M0_hf)**: same bf16 weights loaded via HF
  transformers 5.8.1 + PyTorch 2.10 on MPS (`hf_bf16` loader,
  `device_map=auto` → MPS:0). Result: **81.0 %** (243/300) on the
  old unsorted split. The −4.7 pt gap vs mlx_vlm is under
  investigation (HF attention impl, processor template, MPS numerics).
- **Historical note**: the 85.7 % number (2026-05-15 afternoon) was
  on the unsorted test.jsonl. The 53.0 % (2026-05-14) was on the
  wrong in-distribution val.jsonl. Both superseded by the sorted-
  split result above.

### M4 — `mlx_hybrid_gptq_stable_w4_g128` ✅ (NaN fixed, quality poor)

**v2 (stable GPTQ) — 2026-05-15 evening:**

- Pipeline: hybrid flow + `gptq_stable.py` (our implementation,
  ported from a prior GPTQ implementation). Three fixes over mlx_lm GPTQ:
  1. **desc_act** — columns processed in descending Hessian-diagonal
     order, reducing error amplification.
  2. **Dead-column handling** — zero Hessian diagonal ⇒ zero weight
     column + set H[i,i]=1, preventing singular Cholesky on Gemma 4's
     KV-shared layers.
  3. **Symmetric per-row clipping** — adaptive `clip_sigmas * row_std`
     range instead of per-group min/max asymmetric grid.
- Calibration: 256 samples × 2048 tokens from WikiText-2 train.
- Quant wall time: ~73 min on M5 Pro (Hessian ~68 min, Quantize
  ~5 min for 315 layers, Save ~21 s). **Zero NaN/Inf** — all 315
  layers quantized successfully, none fell back.
- size = **3.9 GB** on disk; effective bits/weight = 5.470.
- species_match = **65.7 %** (197/300); ROUGE-L mean = 0.663
- Drop vs M0: **−22.7 pts** ⚠ — well below the affine variants.
- **Root cause of poor quality**: the stable GPTQ path was designed
  for a ~50M param model at 6-7 bits. Its symmetric per-row scale
  (one scale per entire row) is too coarse for a 4.4B model at 4-bit.
  Per-group asymmetric scale (as in affine M1-M3) preserves much more
  information per group. The Hessian-guided column ordering prevents
  NaN but doesn't compensate for the crude quantization grid.
- **Next step**: hybrid GPTQ with per-group asymmetric scale (combine
  desc_act stability from our impl with per-group grid from mlx_lm).

**v1 (mlx_lm GPTQ) — 2026-05-14 (superseded):**

- Used `mlx_lm.quant.gptq.gptq_quantize` directly. Produced NaN
  scales and Inf biases in 85/317 layers (concentrated on `o_proj`).
  Root cause: three independent bugs in upstream mlx_lm GPTQ when
  applied to Gemma 4 E2B (analysis merged from
  `plan-fix-mlx-gptq.md`, 2026-05-15):

  1. **No desc_act (column reordering).** mlx_lm processes columns
     left-to-right in natural order. GPTQ's error-propagation loop
     amplifies early errors. desc_act
     (`argsort(diag(H), descending)`) processes the most-informative
     columns first, reducing error amplification. The stable GPTQ
     implementation does this; mlx_lm does not.

  2. **No dead-column handling.** If `diag(H)[i] == 0` (a column
     never activated during calibration), Cholesky blows up.
     The stable GPTQ implementation sets `H[dead, dead] = 1` and zeros out the
     corresponding weight columns. mlx_lm has no check. On Gemma 4,
     the KV-shared layers (15-34) have unusual activation patterns
     that produce near-zero Hessian diagonals on `o_proj` —
     exactly where the v1 NaN concentrated.

  3. **Asymmetric quantization grid drifting under error
     propagation.** mlx_lm's `mx.quantize` computes per-group
     `(scale, bias)` for an asymmetric `[0, 2^bits-1]` grid. The
     quantize inner loop then re-uses those scales/biases while
     propagating GPTQ error into neighboring columns — but the
     scales/biases were computed on the ORIGINAL weights, not the
     error-adjusted weights. The mismatch grows across groups and
     eventually produces Inf scales. The stable GPTQ implementation uses
     symmetric per-row clipping with a configurable `clip_sigmas` and updates
     scales only once per row.

- v2 fix shipped as `quantization/src/methods/gptq_stable.py`. All
  three bugs addressed; zero NaN/Inf in the 2026-05-15 run. The
  trade-off (per-row symmetric instead of per-group asymmetric)
  is what costs us 22.7 pt vs M0 — see "next step" above.

### M5 — `mlx_hybrid_awq_w4_g128` ⛔

- Pipeline: hybrid flow + AWQ. The shipped `AWQ_MODEL_CONFIGS`
  registry (`mlx_lm/quant/awq.py:560`) has entries for
  `{llama, mistral, qwen2, qwen3, deepseek_v2, gemma3, gemma3_text}`
  only — no `gemma4_text`. The runner monkey-patches in a copy of the
  `gemma3` entry under the `gemma4_text` key.
- First failure: OOM during `search_best_scale`. AWQ's grid search
  materializes `(n_grid, B, T, H)` tensors. At default n_grid=20 with
  256 × 2048 calibration this asks for 17 GB > Metal's per-buffer
  ceiling of 14.3 GB on M5 Pro. Fixed by dropping to (64 × 512) +
  n_grid=10.
- Second failure: `AttributeError: 'Linear' object has no attribute
  'input_feat'` inside `search_best_clip`. AWQ uses forward hooks to
  attach an `input_feat` activation cache on each linear during
  calibration. The hooks didn't fire on Gemma 4's TransformerLayer
  (likely because the layer's `__call__` signature requires kwargs
  the gemma3 ScaleConfig doesn't supply, e.g. position info or per-
  layer scaling state).
- The doc 08 §5.1 prediction "Need 5-10 line patch to local mlx-lm
  fork (copy gemma3 entry, adjust KV-shared layout)" was the right
  shape but understated: Gemma 4's per-layer-input-projection +
  layer-scalar machinery needs additional ScaleConfig kwargs beyond
  what gemma3 has.
- Verdict: blocked. Either land a proper Gemma 4 entry upstream
  (multi-step) or drop AWQ from the sweep.

### M6 — `mlx_hybrid_dynamic_quant_bpw4_g128_v2` ✅

- Pipeline: hybrid flow + dynamic_quant. target_bpw=4 with
  low=(bits=3, gs=128), high=(bits=4, gs=128). Driven through
  `mlx_vlm.load` + `_LogitsUnwrapper` adapter.
- **v2 changes vs v1** (which timed out at 2 h):
  - Reduced calibration: 64 samples × 512 tokens (v1: 128 × 512)
  - `gradient_checkpoint=True` in `estimate_sensitivities` — trades
    compute for memory, avoids M5 Pro page-compression stalls
  - Result: sensitivity phase completed in **32 min** (16 iters ×
    ~121 s/it, stable rate throughout — no memory-pressure degradation)
- size = **3.2 GB** on disk; effective bits/weight = 4.285 (mix of
  3-bit and 4-bit groups driven by per-layer sensitivity thresholds)
- species_match = **83.3 %** (250/300); ROUGE-L mean = 0.797
- Drop vs M0: **−5.0 pts** 🟡 — close to M1 (83.0 %), slightly
  below M2/M3. Calibration-based sensitivity does not meaningfully
  beat flat affine on this model.
- Eval wall: 146 s on M5 Pro.
- **Historical note (v1)**: the original M6 run (2026-05-14) used
  128 × 512 calibration without `gradient_checkpoint`. Iter rate
  degraded from ~150 s/it to ~200+ s/it as the M5 Pro started
  compressing pages (~70 MB free, 600 MB compressed at iter 14/32).
  Killed at 2 h with no save. `gradient_checkpoint=True` fixes this
  by trading recomputation for peak memory — stable ~121 s/it
  throughout.

### M8 — EoRA post-quant adapters on M2 ✅ (best result)

**EoRA** (arXiv:2410.21271, NVIDIA NeurIPS 2024) is a training-free
post-quantization quality recovery method. It computes a low-rank
adapter (LoRA-shaped A, B pair) per quantized linear by:

1. Collecting calibration covariance X^T X through the quantized model
   (128 samples × 512 tokens from WikiText-2, same as dynamic_quant v2)
2. For each layer: delta = W_bf16 − dequant(W_q)
3. Eigendecompose X^T X → eigenspace scaling that weights the SVD by
   input distribution importance
4. Truncated SVD (via eigendecomposition on the smaller dim for memory
   efficiency) → (A, B) where B @ A ≈ delta

At inference: `y = quantized_linear(x) + x @ A^T @ B^T`
(identical to QLoRA forward pass — mlx-swift-lm already supports this).

**Results (on M2, g64 affine, sorted test.jsonl n=300 seed=0):**

| Rank | Adapter (MB) | Total (GB) | Match | Drop vs M0 | ROUGE-L |
|------|-------------|------------|-------|------------|---------|
| 32   | 105         | 3.5        | 86.3% | −2.0       | 0.823   |
| **64** | **209** | **3.6** | **88.0%** | **−0.3** | **0.839** |
| 128  | 419         | 3.8        | 87.0% | −1.3       | 0.838   |

- **r=64 is the sweet spot**: 88.0% (264/300), only −0.3 pt from M0's
  88.3% ceiling. Within statistical noise for n=300. ROUGE-L rises
  from M2's 0.803 to 0.839 (vs M0's 0.856).
- r=128 regresses slightly vs r=64: the extra rank likely overfits to
  the WikiText calibration distribution (which is text-only, not
  plant-domain). Per-domain calibration (PlantNet calib data) is a
  natural follow-up.
- r=32 is still a strong result: recovers 2.7 of the 4.7 pt gap at
  only 105 MB adapter cost.
- 275 of 315 quantized linears were covered. The 40 skipped layers are
  K/V projections in KV-shared layers 15-34 (no separate bf16 weights
  in the merged model — they share K/V from the global cache).
- Calibration: 28 min. EoRA computation: 23 min. Eval: ~4 min/rank.
  Total pipeline: ~60 min on M5 Pro 32 GB.
- Adapter format: safetensors with `<module_path>.lora_a` (in_dim, r)
  and `<module_path>.lora_b` (r, out_dim) keys, bf16. Compatible with
  mlx-swift-lm's QLoRALinear.

**Implementation:**
- Core algorithm: `src/quantization/src/methods/eora_mlx.py`
- Driver: `src/quantization/scripts/run/eora_post_quant.py`
- Saved adapters: `src/quantization/results/eora_on_m2_g64/`

**Recommended deploy artifact: M2 + EoRA r=64 (3.6 GB, 88.0%).** This
replaces M2 alone (3.4 GB, 83.7%) as the recommended artifact. The
0.2 GB adapter cost buys 4.3 pt of quality back, landing within noise
of the bf16 ceiling.

### M7 — `mlx_hybrid_dwq_w4_g128` ⏳

- Not attempted. DWQ is a training-loop method: it pre-computes
  teacher logits from the bf16 LM, then trains the quantized
  student's scales via KL divergence over (forward + backward + opt
  step) loops. Doc 01b reported a broadcast bug on the 2026-05-13
  mac_mlx_lm run (`Shapes (2,576) and (2,288)` in `dwq.py:113` —
  KV-shared layout mismatch in DWQ's validation-loss path); whether
  hybrid flow fixes that is untested.
- Plumbing cost: mlx_lm.quant.dwq.load_data expects an HF dataset
  path (it uses the tuner's dataset adapters). We'd need to either
  feed it `wikitext` directly or adapt our text_calibration into the
  dwq.iterate_batches format (~30 LOC). Then ~1 h training on this
  hardware.
- Verdict: deferred — low payoff if DWQ's broadcast bug reproduces,
  high plumbing cost otherwise.

| M7 | DWQ | — | — | ⏳ deferred | — |
| **M8b** | **M2 + EoRA r=64** | **3.6 GB** | — | **88.0 %** | **−0.3** |

**M8b (M2 + EoRA r=64) essentially closes the quant gap.** At 88.0%
(264/300) it is within statistical noise of M0's 88.3% ceiling. The
adapter adds only 0.2 GB to M2's 3.4 GB base, staying well under the
4 GB iOS ceiling. No training required — the adapter is computed in
~50 min from calibration data + eigenspace SVD of the weight delta.

**Recommended deploy artifact: M2 + EoRA r=64 (3.6 GB, 88.0%).**
Replaces M2 alone as the recommended artifact. The adapter ships as a
separate safetensors file alongside the quantized model; at load time,
each QuantizedLinear gains a low-rank correction in its forward pass
(QLoRA-shaped: `y += x @ lora_a @ lora_b`). mlx-swift-lm already
supports QLoRALinear, so iOS integration is straightforward.

**Earlier observations still hold:**
- M2/M3 affine variants show the expected g32 > g64 > g128 ordering.
- M4 GPTQ (stable) NaN is fixed but per-row symmetric scale is too
  coarse (−22.7 pt). desc_act + dead-col fixes are sound; the
  quantization grid needs per-group asymmetric.
- M6 calibration-based does not beat affine on its own.

Remaining paths:
1. **Domain-calibrated EoRA** — use PlantNet image+text pairs instead
   of WikiText-only for the X^T X collection. r=128's regression
   (87.0% < r=64's 88.0%) suggests text-only calibration loses
   direction at high rank; domain data may fix this.
2. **M4 v3** — combine desc_act stability with per-group asymmetric
   scale. With EoRA on top, could potentially exceed M0.
3. **EoRA on M1** — r=32 adapter on M1 (3.2 GB) → ~3.3 GB total.
   If quality matches M8b, this is the smallest possible artifact.
4. **B.1 route (GPTQModel on CUDA)** — separate CUDA machine.
5. **Stronger SFT** — vision-tower fine-tuning to raise bf16 ceiling.

## Historical context — mac_mlx_lm round (2026-05-13)

The 2026-05-13 attempt to drive 4 MLX research methods (GPTQ, AWQ, DWQ,
dynamic_quant) through `mlx_lm.load` produced 4 broken outputs because
mlx_lm and mlx_vlm have divergent Gemma 4 model classes (RMSNorm vs
RMSNormZeroShift, `nn.Linear` vs `ScaledLinear` for
`per_layer_model_projection`, KV-shared layer handling, audio config
defaulting). Full post-mortem of the 4 architectural mismatches and
the failed `splice_lm_into_multimodal.py` workaround lives in
[`../general/15-postmortems.md`](../general/15-postmortems.md) §2.

Score sheet of that round:

| Method | Convert | mlx_lm infer | mlx_vlm load | PlantNet eval | Verdict |
|---|---|---|---|---|---|
| `dynamic_quant` | ok | ok (PPL 511) | partial (pad-spam after splice) | BLOCKED | best size (2.99 GB) but iOS path blocked |
| `gptq_w4_g64` | ok | NaN logits | n/a | n/a | unusable (numerical-stability bug under wrong forward) |
| `awq_w4_g64` | FAILED | n/a | n/a | n/a | NYI for gemma4 in mlx_lm 0.31.3 |
| `dwq_w4_g64` | FAILED | n/a | n/a | n/a | upstream broadcast bug |

The corrected approach (hybrid flow: `mlx_vlm.load` first, then
`mlx_lm.quant.*` core on the resulting tree, then `mlx_vlm.utils.save_*`)
is the active design — see [`05-mlx-vlm-design.md`](05-mlx-vlm-design.md)
§4 for the recipe.
